Agent Directive: Fix 400 Bad Request on Property Relationship Filtering
The Problem: Filtering properties by project name triggers a 400 error. The console shows the request is failing at: /api/properties?filters[project][name][$eq]=Downtown Views.

1. Verify the Relation Field Name (Critical)

Go to Strapi Content-Type Builder > Property.

Check the exact name of the relation field linking to Projects.

If the field is named projects (plural), the code must be edited to use filters[projects][name].

If the field is named project_ref or something else, update the filter key to match exactly.

2. Audit Deep Filtering Permissions

In Strapi Settings > Roles > Public, ensure that NOT ONLY is Property findable, but Project is also findable.

Strapi sometimes blocks "Deep Filtering" if the related content type (Projects) isn't also publicly accessible.

3. Fix URL Encoding with qs

The current URL shows Downtown%20Views. While this looks okay, Strapi v4/v5 prefers the qs library for complex nested filters.

Edit the fetcher: Replace manual string concatenation with the qs library to ensure the brackets and operators are encoded in a way Strapi understands.

Example: qs.stringify({ filters: { project: { name: { $eq: 'Downtown Views' } } } })

4. Check for "Draft" Status

Confirm that the Project titled "Downtown Views" is actually Published. If the project is a Draft, filtering for it will often return a 400 or an empty array depending on the Strapi version.

5. Debug the Error Object

In api.ts:27, there is a log for [API Error] Response: 400 Bad Request.

Agent: Expand that object and look at error.details. It will tell you exactly which field is "undefined" or "missing" according to Strapi. Fix based on that specific feedback.
