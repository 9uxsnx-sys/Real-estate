Issue: The filter for properties by project name is failing with a 400 error.
URL in Console: .../api/properties?filters[project][name][$eq]=Downtown Views&populate=*

Task 1: Verify the Relation Path

Check Strapi Schema: Open the properties Content Type in Strapi. Is the relation field exactly named project (singular) or projects (plural)?

Action: Update the filter key to match. If the relation is plural, the filter must be filters[projects][name][$eq]=....

Task 2: Fix URL Encoding

The Problem: The URL contains a space ("Downtown Views"). If the agent is concatenating strings manually, the space might be breaking the request.

Action: Use the qs (Query String) library to stringify the query parameters.

Example Structure: ```javascript
const query = qs.stringify({
filters: {
project: {
name: { $eq: projectName }
}
},
populate: '*'
}, { encodeValuesOnly: true });


Task 3: Audit the "Project" Content Type

Field Check: Does the Project content type actually have a field named name? If the field is named Title or projectName instead, the filter will fail with a 400.

Action: Confirm the exact field name in the Project model and update the filter.

Task 4: Add Defensive Error Handling

In PropertiesListing.tsx, find the filtering logic. If the API returns a 400, log the entire response body from Strapi. Strapi often sends a JSON object explaining why it's a 400 (e.g., "Field not found").
